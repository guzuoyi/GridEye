## 一、项目背景与目标

### 1.1 背景

高速公路交通事件（违停、抛锚、事故、遗撒等）的自动检测是智慧交通的核心需求。传统纯视觉方案存在：
- 误报率高
- 缺乏语义理解能力
- 难以区分「真异常」与「临时停车」等边界场景

### 1.2 目标

构建一套**低成本、可快速验证**的交通事件二次识别系统原型，验证以下核心假设：

| # | 核心目标 | 通过标准 |
|---|---------|---------|
| 1 | "YOLO 初筛 + Qwen 精判"技术路线有效 | 大模型调用减少 ≥ 90%（baseline = 逐帧调用 Qwen） |
| 2 | 车道级定位准确（基于查找表） | 车道号正确，距离误差 < 10% |
| 3 | 双向判断能力 | "事件发生"和"风险消除"均可正确触发 |
| 4 | 全自动标定与维护 | 首日标定成功，查找表可用；后续每日自动更新 |

## 二、整体架构

### 2.1 流水线设计

采用**松耦合流水线**，各模块独立运行，通过 Python dataclass（内部）和 JSON（外部/持久化）标准化接口协同。

```
视频流
  │
  ├─ 抽帧 (1~2 FPS)
  │
  ├─┬─ YOLOv11 目标检测 ──→ 尺寸过滤 ──→ 防闪烁 ──┐
  │ │                                                │
  │ └─ 车道线检测 ──→ 查找表 ──→ 查表定位 ──────────┤
  │                                                  │
  │         ┌────────────────────────────────────────┘
  │         ▼
  │    位置缓存匹配
  │         │
  │         ▼
  │    状态机判定
  │         │
  │    ┌────┼────┬────┐
  │    ▼    ▼    ▼    ▼
  │  IDLE WARNING CONFIRMED CLEARED ──→ MANUAL_REQUIRED
  │         │       │         │
  │         ▼       ▼         ▼
  │      Qwen     Qwen      Qwen
  │    (事件确认)(状态刷新)(风险消除)
  │         │       │         │
  │         └───────┴─────────┘
  │                    │
  │                    ▼
  │              结果输出 + 错误样本归档
```

### 2.2 两路并行处理

| 路径 | 模块 | 功能 |
|------|------|------|
| **路径 A** | YOLOv11n 目标检测 | 检测车辆、行人、两轮车 |
| **路径 B** | OpenCV 车道线检测 | 生成/更新查找表，提供车道级定位 |

两路松耦合：路径 B 可独立标定/更新，不影响路径 A 运行。

### 2.3 模块间接口规范

| 方向 | 格式 | 说明 |
|------|------|------|
| 模块内部传递 | Python `dataclass` | 类型安全，IDE 友好 |
| 跨线程/进程 | JSON 字符串 | 可序列化，便于日志/归档 |
| 持久化存储 | JSON 文件 + NumPy `.npy` | LUT 用 `.npy`，其余用 JSON |

## 三、核心逻辑

### 3.1 目标检测与过滤逻辑

#### 3.1.1 YOLO 检测

| 参数 | 值 |
|------|-----|
| 输入帧率 | 1~2 FPS |
| 原始分辨率 | 1920×1080 |
| YOLO 输入尺寸 | 640×640 |
| 预处理 | 等比缩放（Stretch），无 letterbox/padding |
| 检测类别 | car（轿车）、truck（货车/bus）、person（行人）、two-wheeler（自行车/摩托车） |

- YOLO 输出检测框坐标后 **映射回 1920×1080 原始分辨率**
- 映射公式（等比缩放 Stretch）：
  ```
  x_orig = x_640 × (1920 / 640) = x_640 × 3
  y_orig = y_640 × (1080 / 640) = y_640 × 1.6875
  ```
- 因为输入端直接 resize 到 640×640 不做 padding，坐标映射无需修正

#### 3.1.2 尺寸过滤

> **基准**：原始分辨率 1920×1080 下的边界框高度 `h_box` 和面积 `area_box`。

| 类别 | 最小高度 (px) | 最小面积 (px²) |
|------|:---:|:---:|
| 行人 (person) | 35 | 1000 |
| 轿车 (car) | 22 | 1500 |
| 货车/大巴 (truck) | 28 | 2000 |
| 两轮车 (two-wheeler) | 22 | 1000 |

- 任一条件不满足 → 目标直接丢弃，不进入位置缓存匹配

#### 3.1.3 防闪烁逻辑

> **目的**：避免远处小目标因检测不稳定导致驻留帧数频繁重置。

- 每个目标维护 **最近 10 帧** 的尺寸有效性记录（布尔队列）
- **最近 10 帧中 ≥ 7 帧有效** → 认定为稳定目标，正常累加驻留帧数
- **有效帧 < 7** → 驻留帧数重置为 0

**防闪烁的「尺寸有效」判定**：沿用 §3.1.2 的类别阈值 × **0.8**（放宽系数）：

| 类别 | 防闪烁最小高度 (px) | 防闪烁最小面积 (px²) |
|------|:---:|:---:|
| 行人 (person) | 28 | 800 |
| 轿车 (car) | 17.6 | 1200 |
| 货车/大巴 (truck) | 22.4 | 1600 |
| 两轮车 (two-wheeler) | 17.6 | 800 |

### 3.2 车道线与查找表逻辑

#### 3.2.1 查找表概述

- **定义**：二维数组 `LUT[y][x] = { lane: int, distance: float }`
- **尺寸**：1080 行 × 1920 列（对应原始分辨率每像素）
- **查表复杂度**：O(1)
- **车道号约定**：从图像左侧（道路外侧）起算，**0 = 硬路肩/应急车道**，1 = 第一行车道，2 = 第二行车道，依次递增。仅行车道（≥1）参与事件判定。

#### 3.2.2 首日一键自动标定

| 步骤 | 操作 | 输入 | 输出 |
|------|------|------|------|
| Step 1 | 相机内参标定 | 棋盘格标定板（A3，多角度放置） | 内参矩阵 K、畸变系数 dist |
| Step 2 | IPM 矩阵与外参标定 | 车道线特征（长度、标准宽度 3.75m、消失点） | 俯仰角、偏航角、离地高度、IPM 单应性矩阵 H |
| Step 3 | 查找表生成 | H 矩阵 + 车道线方程 | 每个 (u,v) → (车道号, 纵向距离) |

- Step 2 无需额外标定板，纯靠车道线几何约束求解
- 棋盘格仅在 Step 1 需要

#### 3.2.3 日常自动维护

- **触发**：每日定时执行（建议凌晨 3:00）
- **流程**：
  1. 抓取当前一张清晰图像
  2. IPM 变换 → 鸟瞰图 → 预处理 → Hough 变换检测 → 聚类拟合车道线方程
  3. 基于**不变的 IPM 矩阵 H** + 最新车道线方程，重新生成查找表
  4. **原子替换**：新 LUT 数组生成后，`active_lut = new_lut`（Immutable Snapshot）
- **隐含假设**：相机位姿不变（H 不变），仅车道线可能因磨损而变化

#### 3.2.4 查找表并发安全

- 运行时查表为**只读**，每日维护为**写入**
- 策略：**Immutable Snapshot**——生成完整新 LUT 后，原子替换全局引用
- Python GIL 保证引用赋值是原子的，无需额外加锁

### 3.3 位置缓存与状态管理逻辑

#### 3.3.1 位置匹配（不依赖目标 ID）

- 取检测框**中心点** `(cx, cy)`
- 与缓存中所有 **未过期** 目标比对
- **匹配条件**：类别相同 && 中心点欧氏距离 < **40 px**（基于 1920×1080）
- 匹配成功 → 更新该位置记录；匹配失败 → 创建新位置记录
- 边界处理：多个缓存匹配同一检测 → 选距离最近者

#### 3.3.2 位置缓存数据结构

```
PositionRecord {
    id: str
    class: str                        // car / truck / person / two-wheeler
    center_x: float, center_y: float  // 1920×1080
    consecutive_frames: int
    last_seen_time: float             // 秒
    state: enum { IDLE, WARNING, CONFIRMED, CLEARED, MANUAL_REQUIRED }
    state_timestamps: {
        idle: float, warning: float, confirmed: float,
        cleared: float, manual_required: float
    }
    llm_result: {
        is_anomaly: bool, event_type: str,
        confidence: float, reasoning: str
    }
    lane_number: int
    longitudinal_distance: float      // 米
    size_valid_history: bool[10]
}
```

#### 3.3.3 状态机

```
                  ┌───────────────┐
                  │     IDLE      │  ← 新目标首次出现
                  └───────┬───────┘
                          │ 连续驻留 ≥ 5 帧
                          ▼
                  ┌───────────────┐
          ┌───────│   WARNING     │───────┐
          │       └───────┬───────┘       │
          ▼               │               ▼
   Qwen 判定正常    Qwen 判定异常
   或目标消失
          │               │
          ▼               ▼
      ┌───────┐   ┌───────────────┐
      │ IDLE  │   │  CONFIRMED    │
      └───────┘   └───────┬───────┘
            ▲             │ 连续 3 秒未命中
            │             ▼
            │     ┌───────────────┐
            │     │   CLEARED     │
            │     └───┬───┬───────┘
            │         │   │
            │  消除   │   │ 未消除
            │         │   │
            │   ┌─────┘   ▼
            │   │ ┌──────────────────┐
            └───┘ │ MANUAL_REQUIRED  │
                  └──────┬───────────┘
                         │ 人工确认
                         ▼
                     ┌───────┐
                     │ IDLE  │
                     └───────┘
```

#### 3.3.4 状态迁移规则（9 条）

| # | 当前状态 | 触发条件 | 下一状态 | 说明 |
|---|---------|---------|---------|------|
| 1 | IDLE | 连续驻留 ≥ 5 帧 | WARNING | 触发 Qwen 事件确认 |
| 2 | WARNING | Qwen 判定异常 | CONFIRMED | 事件确认 |
| 3 | WARNING | Qwen 判定正常 / 目标消失 ≥ 3s | IDLE | 误报或短时停留 |
| 4 | CONFIRMED | 目标消失 ≥ 3 秒 | CLEARED | 触发 Qwen 风险消除确认 |
| 5 | CONFIRMED | 持续存在 > 30 秒 | CONFIRMED | 周期性刷新状态 |
| 6 | CLEARED | Qwen 确认风险消除 | IDLE | 归档，恢复正常 |
| 7 | CLEARED | Qwen 判定风险未消除 | MANUAL_REQUIRED | 保留告警，等待人工 |
| 8 | MANUAL_REQUIRED | 人工确认风险消除 | IDLE | 恢复正常 |
| 9 | MANUAL_REQUIRED | 人工确认仍存在 | CONFIRMED | 恢复事件确认 |

#### 3.3.5 过期清理

| 状态 | 条件 | 动作 |
|------|------|------|
| IDLE | 连续 3 秒未命中 | 直接删除缓存 |
| WARNING | 连续 3 秒未命中 | 直接删除缓存 |
| CONFIRMED | 连续 3 秒未命中 | 转为 CLEARED |
| CLEARED | Qwen 处理完毕 | 删除或回 IDLE |
| MANUAL_REQUIRED | 超时 300s 未获人工确认 | 记录告警日志，保持状态 |

### 3.4 大模型调用逻辑

#### 3.4.1 大模型选型

| 项目 | 选型 |
|------|------|
| 模型 | Qwen3.5-9B，LM Studio 本地部署 |
| 调用方式 | OpenAI 兼容 API：`POST http://localhost:1234/v1/chat/completions` |
| 视觉输入 | OpenAI Vision 格式：图片 base64 编码 → `image_url` 嵌入 messages |
| SDK | `openai` Python SDK，`base_url=http://localhost:1234/v1`, `api_key=lm-studio`（占位） |
| 输入图像尺寸 | 448×448（目标裁剪后缩放） |
| 端到端延迟预算 | **≤ 5 秒**（从抽帧到状态机输出，不含人工响应时间） |

#### 3.4.2 API 调用格式（OpenAI Vision）

```python
from openai import OpenAI

client = OpenAI(base_url="http://localhost:1234/v1", api_key="lm-studio")

response = client.chat.completions.create(
    model="qwen3.5-9b",  # LM Studio 中加载的模型名
    messages=[
        {"role": "system", "content": "你是高速公路交通事件分析专家..."},
        {"role": "user", "content": [
            {"type": "text", "text": "上下文: {类别: 轿车, 车道号: 1, ...}"},
            {"type": "image_url", "image_url": {"url": "data:image/png;base64,..."}}
        ]}
    ],
    temperature=0.1,
    max_tokens=512
)
```

#### 3.4.3 三种调用场景

**场景一：事件确认**

| 项目 | 内容 |
|------|------|
| **触发条件** | IDLE → WARNING（连续驻留 ≥ 5 帧） |
| **输入** | 目标裁剪图（448×448, base64）+ 结构化上下文 JSON |
| **输出** | `{ is_anomaly: bool, event_type: str, confidence: float, reasoning: str }` |
| **后续** | 异常 → CONFIRMED；正常 → IDLE |

**场景二：状态刷新**

| 项目 | 内容 |
|------|------|
| **触发条件** | CONFIRMED 持续超过 30 秒（周期性） |
| **输入** | 目标裁剪图 + 当前上下文 + 历史判定结果 |
| **输出** | `{ status_changed: bool, updated_event_type: str }` |
| **后续** | 更新事件记录 |

**场景三：风险消除确认**

| 项目 | 内容 |
|------|------|
| **触发条件** | CONFIRMED 目标消失 ≥ 3 秒 |
| **输入** | 历史事件信息（纯文本，无图片）：类别、车道号、距离、事件类型、确认时间、消失时间 |
| **输出** | `{ risk_cleared: bool, need_cleanup: bool, confidence: float, reasoning: str }` |
| **后续** | 消除 → IDLE；未消除 → MANUAL_REQUIRED |

#### 3.4.4 调用频率预估

- 假设 2 FPS，单画面 5 辆车，其中 1 辆触发 WARNING
- 该车：事件确认 1 次 + 每 30s 刷新 1 次 + 风险消除 1 次
- **Qwen 调用/事件 ≈ 3~5 次**
- **H1 基线（逐帧调用）**：2 FPS × 60s = 120 次/min → 减少 ≥ 95%

## 四、数据闭环逻辑

### 4.1 错误样本自动归档

```
archive/
├── event_confirmation/          # 事件确认相关
│   ├── TP/                      # 正确识别
│   ├── FP/                      # 误报
│   └── FN/                      # 漏报
```

## 五、坐标与分辨率约定

| 环节 | 分辨率/尺寸 | 说明 |
|------|:---:|------|
| 原始视频帧 | 1920×1080 | 摄像头采集原始图像 |
| YOLO 输入 | 640×640 | 等比缩放（Stretch），无 letterbox/padding |
| YOLO 输出映射 | 1920×1080 | `x_orig = x_640 × 3`, `y_orig = y_640 × 1.6875` |
| 查找表索引 | 1920×1080 | LUT[y][x]，逐像素构建 |
| 尺寸过滤阈值 | 基于 1920×1080 | 高度/面积阈值均以原始分辨率为准 |
| 防闪烁尺寸阈值 | 基于 1920×1080 | 过滤阈值 × **0.8** |
| 匹配距离阈值 | 40 px | 基于 1920×1080，其他分辨率按比例缩放 |
| Qwen 输入裁剪图 | 448×448 | 从原图裁剪目标区域后缩放 |
| 端到端延迟预算 | ≤ 5 秒 | 从抽帧到状态机输出 |

> **关键原则**：所有阈值、匹配距离、查找表均以 1920×1080 为基准。更换分辨率需等比缩放。

## 七、技术选型汇总

| 组件 | 选型 | 说明 |
|------|------|------|
| 目标检测 | YOLOv11n | nano 版，CPU/GPU 均可 |
| 车道线检测 | OpenCV (IPM + Hough) | 传统视觉，零额外模型 |
| 内参标定 | OpenCV 棋盘格标定 | 成熟稳定 |
| 外参标定 | 车道线几何约束自动求解 | 无需额外标定板 |
| 查找表 | NumPy 结构化数组 | `dtype=[('lane','i4'),('dist','f4')]` |
| 大模型 | Qwen3.5-9B | LM Studio 本地部署 |
| 推理框架 | LM Studio（OpenAI 兼容 `/v1/chat/completions`） | 默认 `http://localhost:1234` |
| API 调用 | `openai` Python SDK | `base_url=http://localhost:1234/v1` |
| 模块接口 | Python dataclass（内部）+ JSON（外部） | 类型安全，易于调试 |
| 语言 | Python 3.10+ | |

## 八、关键设计原则总结

| 原则 | 说明 |
|------|------|
| **YOLO 粗筛，Qwen 精判** | 各司其职，大模型仅处理可疑事件 |
| **不依赖 ID，只依赖位置** | 彻底规避低帧率 ID 跳变 |
| **查表替代实时计算** | 预计算 (u,v)→(车道号, 距离)，运行时 O(1) |
| **首日一键自动标定** | 棋盘格内参 + 车道线外参，全自动 |
| **每日自动维护** | 车道线检测 + LUT 原子更新 |
| **松耦合流水线** | YOLO、车道线、Qwen 相互独立 |
| **双向判断** | 事件发生 + 风险消除，两次大模型确认 |
| **分辨率统一** | 所有阈值/距离/LUT 基于 1920×1080 |
| **Immutable Snapshot** | LUT 引用原子替换，运行时无锁 |
| **OpenAI 兼容 API** | LM Studio 提供标准 `/v1/chat/completions` |
| **5 秒延迟预算** | 端到端 ≤ 5s，实验阶段优先准确率 |
